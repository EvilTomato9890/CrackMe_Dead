# Sanya Crackmes

В этой папке лежат два DOS `.COM` crackme и рабочие файлы для их эксплуатации:

- `simple.COM` + `simple_exploit.bin`
- `hard.COM` + `hard_exploit.bin`

Ниже описаны структура кода и практический способ взлома для каждого файла.

## Допущение

Для удобства далее в тексте принято следующее:
- смещения из `objdump` ниже даны как смещения внутри файла
- адрес в памяти DOS равен `file_offset + 0100h`

Пример:
- file offset `0x121` превращается в runtime offset `0x221`

## Общая схема

Обе программы устроены похоже:

1. `jmp` перепрыгивает встроенные данные к основному коду
2. через `int 21h`, `AH=09h` печатается приглашение
3. через `int 21h`, `AH=0Ah` читается ввод
4. считается хэш встроенной строки
5. считается хэш пользовательского ввода
6. оба хэша сравниваются
7. проверяется однобайтовый флаг доступа
8. печатается `Access granted :)` или `Access denied >:(`

Хэш-функция в обеих программах одинаковая:

- стартовое значение `AX = 0`
- для каждого байта: байт прибавляется к `AX`, затем результат умножается на `7`

Встроенная строка для сравнения:

`ronaldo better than coca cola$`

## simple.COM

### Структура кода

Важные смещения внутри файла:

- `0x000`: `jmp 0x192`
- `0x003`: пароль
- `0x121`: заголовок DOS-буфера ввода
- `0x123`: первый байт пользовательского ввода
- `0x141`: однобайтовый флаг доступа
- `0x142`: сохраненный хэш встроенной строки
- `0x144`: сохраненный хэш пользовательского ввода
- `0x146`: `Access granted :)`
- `0x16C`: `Access denied >:(`
- `0x192`: точка входа в основной код

Ключевые переходы:

- `0x199`: чтение ввода
- `0x19F`: сравнение двух хэшей
- `0x1A8`: если хэши равны, в `0x141` записывается `1`
- `0x1AD`: проверка флага
- `0x1D9`: ветка успеха
- `0x1EA`: ветка провала

Фрагмент проверки:

```asm
019f: a1 42 02           mov ax,[0242h]
01a2: 3b 06 44 02        cmp ax,[0244h]
01a6: 75 05              jne 01adh
01a8: c6 06 41 02 01     mov byte ptr [0241h],1
01ad: 80 3e 41 02 00     cmp byte ptr [0241h],0
01b2: 74 36              je 01eah
01b4: eb 23              jmp 01d9h
```

Здесь  видно, что после сравнения хэшей программа просто выставляет флаг в `1`, а потом отдельно проверяет этот же флаг.

Вырезка из области данных:

```text
0120: 00 FF 00 00 00 00 00 00 00 00 00 00 00 00 00 00
0130: 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
0140: 00 00 00 00 00 00 41 63 63 65 73 73 20 67 72 61
```

Здесь видно следующее

- `0x121 = FF` это максимальная длина DOS-буфера
- `0x122` это байт фактической длины ввода
- `0x123` это начало пользовательских данных
- `0x141` это сам флаг доступа
- `0x142..0x145` это два слова с хэшами
- `0x146` это начало строки `Access granted :)`

### Какая уязвимость

DOS-буфер начинает писать пользовательские данные с `0x123`.
Флаг доступа лежит по адресу `0x141`.

Расстояние:

- `0x141 - 0x123 = 0x1E` байт

Это значит:

- 30 байт просто заполняют промежуток
- 31-й байт уже перетирает сам флаг

Программа проверяет только одно условие: нулевой флаг или ненулевой. Если записать туда любой ненулевой байт, финальная ветка пойдет в `Access granted` даже при неверном хэше.

### Эксплойт

Текущий exploit-файл:

- `simple_exploit.bin`

Форма загрузки:

- 31 байт `0x41` (`A`)
- завершающий `0x0D`

Как это работает:

- байты `0..29` доходят до флага
- байт `30` переписывает флаг значением `0x41`
- финальный `0x0D` корректно завершает ввод для DOS-функции

## hard.COM

### Структура кода

Важные смещения внутри файла:

- `0x000`: `jmp 0x192`
- `0x021`: однобайтовый флаг доступа
- `0x122`: заголовок DOS-буфера ввода
- `0x124`: первый байт пользовательского ввода
- `0x146`: `Access granted :)`
- `0x16C`: `Access denied >:(`
- `0x192`: основная логика после ввода
- `0x1A8`: `mov byte ptr [0x121], 1`
- `0x1AD`: сравнение флага с нулем
- `0x1B2`: переход в успех, если флаг не нулевой
- `0x1BA`: процедура успеха
- `0x1D2`: процедура отказа

Вырезка из области данных:

```text
0020: 24 00 0D 00 00 00 00 00 00 00 00 00 00 00 00 00
0120: 00 00 A0 00 00 00 00 00 00 00 00 00 00 00 00 00
```

Здесь видно следующее:

- `0x021 = 00` это байт флага доступа
- `0x122 = A0` это максимальная длина DOS-буфера
- `0x123` это байт фактической длины ввода
- `0x124` это начало пользовательских данных

По этой вырезке хорошо видно отличие от `simple.COM`: флаг и буфер лежат далеко друг от друга.

### Какая уязвимости

Расстояния:

- точка начала кода после ввода: `0x192`
- начало пользовательских данных: `0x124`
- `0x192 - 0x124 = 0x6E`

Блок сравнения начинается чуть дальше:

- `0x199 - 0x124 = 0x75`

Этого достаточно, чтобы перезаписать кусок логики проверки.

### Структура эксплойта

Текущий exploit-файл:
- `hard_exploit.bin`

Он устроен так:
1. `0x00..0x1C`: 29 байт `0x61` (`a`)
2. `0x1D..0x6D`: байты, сохраняющие нужную структуру данных и строк
3. `0x6E..0x80`: 19 байт `0x90` (`NOP`)
4. `0x81..0x83`: байты `05 05 0D`

Ключевое переписывание попадает в диапазон `0x192..0x1A7`.

Изначально там находится такой код:

```asm
0192: b4 0a              mov ah,0Ah
0194: ba 22 02           mov dx,0222h
0197: cd 21              int 21h
0199: e8 63 00           call 01ffh        ; вызвать расчет хэша встроенной строки
019c: e8 70 00           call 020fh        ; вызвать расчет хэша пользовательского ввода
019f: a1 42 02           mov ax,[0242h]
01a2: 3b 06 44 02        cmp ax,[0244h]
01a6: 75 05              jne 01adh
01a8: c6 06 21 01 01     mov byte ptr [0121h],1
01ad: 80 3e 21 01 00     cmp byte ptr [0121h],0
01b2: 75 06              jne 01bah
```

После переписывания эффект такой:

```asm
0192: 90                 nop
...
01a4: 90                 nop
01a5: 05 05 0d           add ax,0D05h
01a8: c6 06 21 01 01     mov byte ptr [0121h],1
01ad: 80 3e 21 01 00     cmp byte ptr [0121h],0
01b2: 75 06              jne 01bah
```

Тоесть эксплойт просто удаляет все проверки строки, после чего выполнение доходит до штатной инструкции, которая сама выставляет флаг доступа в `1`.

Дальше программа уже идет по своей обычной успешной ветке:

- флаг становится ненулевым
- проверка проходит
- печатается `Access granted :)`

## Генерация exploit-файлов

В папке лежат два Python-скрипта, которые заново создают ввод:

- `make_simple_exploit.py`
- `make_hard_exploit.py`

Запуск:

```powershell
python .\make_simple_exploit.py
python .\make_hard_exploit.py
```

## Короткое сравнение

- `simple.COM`: переписывается ближайший флаг
- `hard.COM`  : переписывается участок кода, который должен привести программу в success-ветку
